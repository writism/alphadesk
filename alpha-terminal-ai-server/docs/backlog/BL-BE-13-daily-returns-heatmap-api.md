# BL-BE-13

**Backlog Type**  
API / Data / Cost-control Backlog

**Backlog Title**  
일별 등락 히트맵용 시계열 API — 무료·저비용 데이터 소비 최소화

> **구현 상태:** 배치 API·bucket 페이로드·인메모리 캐시·부분 실패는 구현됨. 공급자별 상세는 **BL-BE-14**(한국 `data.go.kr` + 미국 Twelve Data)를 따른다.

---

## 1. 배경 / 문제

- 프론트에서 **실시간 시세 폴링** 대신, **최근 약 4~6주(또는 1달)** 구간의 **일별 등락**을 깃허브 기여도 그래프와 유사한 **격자(주×요일)** 로 표현하려 한다.
- 외부 시세·캔들 API는 **무료 티어에 분당/일당 호출 한도**가 있어, 종목 수 × 화면 수만큼 **중복 호출**이 나면 금방 한도에 걸린다.
- 브라우저가 외부 API를 직접 두드리면 **키 노출·CORS** 문제가 있으므로, **백엔드 단일 진입점 + 캐시**가 필요하다.

## 3. 무료(저비용) 데이터 소비 최소화 — 조사·설계 방향

### 3.1 외부 소스 후보(구현 전 재확인)

| 소스 | 용도 | 비고 |
|------|------|------|
| **Finnhub** `stock/candle` 등 | 일봉 시계열 | 이미 서버에 키·연동 경험 있음. **한국 종목 커버리지·심볼 규칙**은 문서/실측으로 검증. |
| **기타** | 국내 전용 | Finnhub로 불충분 시 **별도 어댑터**(증권사 Open API 등)는 **별 백로그**로 분리 가능. |

*정확한 분당/일당 한도는 **공급자 최신 문서**를 기준으로 본 백로그 구현 시 표에 갱신한다.*

### 3.2 호출 수를 줄이는 패턴

1. **배치 엔드포인트**  
   - 예: `GET /stocks/daily-returns-heatmap?symbols=005930,AAPL&weeks=6`  
   - 서버가 심볼별로 외부 1회(또는 캐시 히트 0회)만 유도.
2. **서버 측 캐시**  
   - 키: `(provider, symbol, resolution=D, period_end_bucket)` 등.  
   - **TTL**: 일봉 기반이면 **15분~6시간** 또는 **장 마감 후 1회 갱신** 수준으로 시작(제품 합의).  
   - 동일 사용자·다른 사용자 간 **캐시 공유**로 트래픽 폭주 완화.
3. **중복 제거**  
   - 한 요청 안에 동일 `symbol` 중복 시 **한 번만** upstream 호출.
4. **기간 고정**  
   - 기본 `weeks=6`(약 30거래일 근처) 등 **상한**을 두어 candle `from`~`to` 범위를 고정.

### 3.3 페이로드를 줄이는 패턴 (시각 정보는 최대한 유지)

히트맵에 필요한 것은 **“날짜별 방향 + 강도(단계)”** 이다. 옵션:

| 방식 | 내용 | 장점 |
|------|------|------|
| **A. 압축 시퀀스** | 거래일 순서대로 `-2,-1,0,1,2`(하락 진함~상승 진함) 정수 배열 + **시작 영업일 ISO** 한 개 | 바이트 수 최소에 가깝다. |
| **B. (날짜, bucket)** | `[["2025-03-10",2],["2025-03-11",-1],…]` | 디버깅 쉬움, 약간 무거움. |
| **C. Run-length** | 연속 같은 bucket 묶기 | 횡보 구간이 길면 유리. |

**권장:** V1은 **B**로 계약을 고정하고, 한도 이슈가 보이면 **A**로 최적화(프론트 디코더만 추가).

**bucket 정의(예시):**  
- `0`: 보합 또는 변동률 절댓값 &lt; ε  
- `±1, ±2`: 전일 대비 등락률 구간을 **분위수 또는 고정 임계치**로 나눔(백엔드에서만 계산해 프론트는 색만 매핑).

### 3.4 시각적 정보를 “무료 데이터 적게”로 늘리는 트릭

- **강도 단계**만 서버에서 주고, **색 농도·테두리**는 프론트 Tailwind로 매핑(추가 API 없음).
- **한 줄 메타**만 추가: `up_days`, `down_days`, `flat_days`, `as_of`(마지막 일봉 기준일) — 텍스트 요약으로 인지량 증가.
- 격자 **레이아웃(주×요일)** 은 **프론트가 `as_of`와 거래일 리스트만으로 계산**해 서버가 “좌표”를 보내지 않아도 됨(서버는 **영업일별 bucket** 만 책임).

## 4. API 초안 (합의 후 OpenAPI 반영)

### 4.1 요청

- `GET /stocks/daily-returns-heatmap`  
- Query: `symbols` (반복 또는 콤마 구분), `weeks` (기본 6, 상한 예: 13)

### 4.2 응답 (예시 스키마)

```json
{
  "as_of": "2025-03-24",
  "weeks": 6,
  "items": [
    {
      "symbol": "005930",
      "market": "KOSPI",
      "series": [["2025-02-10", 1], ["2025-02-11", -2], ["2025-02-12", 0]],
      "summary": { "up": 12, "down": 8, "flat": 2 }
    }
  ],
  "errors": [{ "symbol": "XXXX", "code": "NO_DATA", "message": "..." }]
}
```

- `series`는 **거래일만** 포함(휴장일 생략). 프론트가 깃허브형 격자에 **빈 칸**으로 채움.

## 6. 비기능 / 리스크

- **약관·지연 시세**: 응답에 데이터 출처·지연 가능성은 프론트 면책과 맞출 것.
