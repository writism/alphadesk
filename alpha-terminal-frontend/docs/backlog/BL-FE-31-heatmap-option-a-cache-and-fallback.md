# BL-FE-31

**Backlog Type**  
UX / Data Fetch / Performance Backlog

**Backlog Title**  
권장안 A 히트맵 데이터 소비 최적화 — 페이지 간 캐시 재사용과 조용한 폴백

---

## 1. 배경 / 문제

- **BL-FE-30**으로 히트맵 UI 자체는 붙었지만, 공급자 이슈가 있으면 카드 안에 아무것도 안 보여 사용자가 원인을 파악하기 어렵다.
- 히트맵은 **실시간 데이터가 아니므로** 페이지마다 매번 새로 fetch 할 필요가 없다.
- 권장안 A(`data.go.kr` + Twelve Data)에서는 **백엔드가 배치 API**를 제공하므로, 프론트는 **중복 요청을 줄이는 캐시**에 집중하는 편이 맞다.

---

## 2. 목표 (Success Criteria)

| ID | 기준 |
|----|------|
| SC-1 | 대시보드·관심종목 페이지는 동일 심볼 집합에 대해 **짧은 시간 내 재마운트 시 즉시 캐시된 heatmap 응답**을 재사용한다. |
| SC-2 | 초기 로딩 중에는 히트맵 자리의 높이가 크게 흔들리지 않는다. |
| SC-3 | `items[]`가 비어도 카드 레이아웃은 유지되고, 요약 카드 본문을 가리지 않는다. |
| SC-4 | 프론트는 공급자 세부사항(Finnhub / Twelve Data / data.go.kr)을 알지 않고, **기존 heatmap 응답 계약**만 소비한다. |

---

## 3. 구현 방향

1. **모듈 레벨 메모리 캐시**
   - 키: 정렬된 `symbols[]` + `weeks`
   - TTL: 일봉 데이터 특성상 **10분~30분** 수준
2. **배치 요청만 유지**
   - 카드별 fetch 금지
3. **조용한 폴백**
   - `errors[]`가 있어도 해당 심볼만 히트맵 숨김
   - 카드 본문은 그대로 표시
4. **향후 확장**
   - 필요 시 React Query/SWR로 교체 가능하지만, 현재는 로컬 모듈 캐시로 충분
